## ЁЯФ┤ Critical (Pi Stability)

### 1. Replace Playwright with Requests + BeautifulSoup
- **Why:** Playwright launches headless Chromium тАФ ~500MB+ RAM on a Pi causes OOM kills
- **Where:** `app/services/sourcing.py`, `app/ml/scraper.py`
- **Action:** Replace all Playwright browser calls with `requests` (with robust user-agent rotation) + `BeautifulSoup` for HTML parsing
- **If JS rendering is strictly needed:** Use a separate lightweight service, not inline browser launches
- **Test:** Verify all vendor scraping (The Malt Miller, Get Er Brewed) still works

### 2. Move Scraping to Background Jobs
- **Why:** Synchronous scraping blocks the Flask server, freezing the dashboard
- **Where:** `app/services/sourcing.py`, `app/api/routes.py`
- **Action:** Use `APScheduler` or similar тАФ API returns a "Job Started" ID immediately, results polled or pushed via SocketIO
- **Bonus:** Eliminates `time.sleep()` blocking calls

### 3. Replace Pandas with Lightweight Alternatives
- **Why:** Pandas + NumPy consume significant RAM for small homebrew datasets
- **Where:** Any service importing pandas
- **Action:** Use standard Python dicts/lists/comprehensions. For stats, use the `statistics` stdlib module

## ЁЯЯа Important (Security)

### 4. Add Authentication
- **Why:** Zero auth тАФ anyone on LAN has full control including config overwrite
- **Where:** `app/main.py`, `app/api/routes.py`
- **Action:** Add Flask HTTP Basic Auth or simple token auth to all API routes
- **Minimum:** Environment variable for API token, checked via decorator

### 5. Fix SSRF Risk in Sourcing
- **Why:** `/api/sourcing/compare-by-tag/<tag>` passes user input to external web requests
- **Where:** `app/api/routes.py`, `app/services/sourcing.py`
- **Action:** Validate and sanitise tag input. Restrict outbound requests to known vendor domains only

## ЁЯЯб Quality (Code Health)

### 6. Fix Exception Handling
- **Why:** Broad `except Exception` everywhere hides real errors
- **Where:** All services
- **Action:** Catch specific exceptions (e.g., `requests.Timeout`, `KeyError`, `InfluxDBError`). Log with context. Return meaningful error responses

### 7. Clean Up Lazy Imports
- **Why:** Heavy imports inside functions mask dependency issues and cause first-request lag
- **Where:** Throughout `app/services/`
- **Action:** Move imports to top of file. If optional, use proper feature flags or try/except at module level

### 8. Improve Testing
- **Why:** Minimal coverage, fragile `sys.modules` mocking
- **Where:** `tests/`
- **Action:** 
  - Switch to `pytest` with `unittest.mock.patch`
  - Add fixtures for InfluxDB mocking
  - Cover core services: sourcing, tilt_monitor, water_chemistry, alerts
  - Add integration test for the API routes

## ЁЯЯв Optimisation (Performance & DevOps)

### 9. GitHub Actions Build Pipeline
- **Why:** Building Docker images (especially Next.js) on the Pi is slow and may OOM
- **Action:** Set up GitHub Actions workflow:
  - Build `linux/arm64` images on GitHub runners
  - Push to GHCR or Docker Hub
  - Pi just pulls pre-built images
- **Template:**
  ```yaml
  - uses: docker/build-push-action@v5
    with:
      platforms: linux/arm64
      push: true
  ```

### 10. Static Export for Frontend
- **Why:** Full Next.js SSR with Node.js runtime is heavy for a Pi
- **Where:** `web/next.config.ts`
- **Action:** Add `output: 'export'` to Next.js config. Serve static files via Nginx or Flask. Remove Node.js runtime dependency on Pi

### 11. Config Caching
- **Why:** InfluxDB queried for every config read (API keys, settings) тАФ unnecessary overhead
- **Where:** `app/services/` (wherever config is read from InfluxDB)
- **Action:** Cache config in memory or local JSON file. Refresh on explicit update only

### 12. Optimise Docker Compose
- **Where:** `docker-compose.yml`
- **Action:**
  - Use pre-built images instead of `build:` on the Pi
  - Add `platform: linux/arm64` explicitly
  - Consider multi-stage Dockerfile to reduce image size
